## «Эй, ухнем»

Представьте галеру. Большую, корпоративную. Галера плывёт. Куда именно — вопрос философский, но быстро и уверенно, потому что девиз простой и понятный: **BE CALM RUN FAST**.

За вёслами — DevOps-инженеры. Их мало, потому что это «дорого», а те, что есть, и так «справляются». От этого они недобрые, уставшие и слегка саркастичные, что видно даже по коммит-месседжам.

То тут, то там периодически падает очередной environment.

За последние полгода он не то чтобы переставал ломаться. Ломается ровно так же, как в прошлый раз. И в позапрошлый. Потому что образ, на котором всё держится, собирал Данила пару лет назад — с душой, но без намёка на повторяемость. Теперь Данила начальник. Мы рады за него, но образ от этого свежее не стал.

Документации нет. Есть Вася, которому один раз показали. Теперь Вася — оракул:
«Там вроде патч накатить… кажется… если помню… хотя не помню».

Сверху менеджер шлёт в мессенджер:
«Нужна новая среда. Желательно позавчера».

Снизу в Jira прилетает от разработчика:
«А почему у меня не ставится библиотека, которая у Жени ставится?».

Сбоку появляется security и засыпает мейлами:
«Когда вы уже обновите базовый образ? У вас там CVE ещё с прошлого квартала».

И понимаешь: живёшь не в инфраструктуре, а в **фольклоре**.

## Три слоя: философия матрёшки

Ключевая идея — разбить монолитный образ на **три независимых слоя**. Каждый строится
поверх предыдущего, у каждого — своя зона ответственности и свой владелец.

> *Монолит удобен ровно до «давай просто чуть оптимизируем».*

```text
  Владелец           Слой                             Содержимое                         Артефакт
  ────────────────────────────────────────────────────────────────────────────────────────────────

  👩‍💻 Команды      ┌─────────────────────────────────────────────────────────┐
  разработки        │  ③  org-golden                                         │   ──►  .box
                    │     Docker, kubectl, компилеры, Python venv,           │
                    │     IDE helpers, team-specific SDK                      │
                    └───────────────────────┬─────────────────────────────────┘
                                            │  packer build (vagrant provisioner)
                                            │  Ansible: playbook/golden.yml
                                            ▼
  🏗️ Инфра         ┌─────────────────────────────────────────────────────────┐
  команда          │  ②  org-base                                           │   ──►  .box
                    │     Corp CA, internal DNS/NTP, proxy settings,         │
                    │     monitoring agent, inventory agent,                  │
                    │     internal apt/yum repos                              │
                    └───────────────────────┬─────────────────────────────────┘
                                            │  packer build (vagrant provisioner)
                                            │  Shell scripts (OS-specific + common)
                                            ▼
  🔒 Безопас-      ┌─────────────────────────────────────────────────────────┐
  ность            │  ①  vendor-base                                        │   ──►  .box
                    │     Pristine OS from official ISO                      │
                    │     Kickstart / Cloud-Init                              │
                    │     BIOS legacy  ──or──  UEFI + efivars.fd             │
                    │     NO corp config, NO internal repos                  │
                    └─────────────────────────────────────────────────────────┘
                                            ▲
                                            │  packer build (QEMU/KVM)
                                       [ ISO файл ]
```

**vendor-base** — нетронутая ОС прямо из ISO. Ноль корпоративного. Ноль агентов.
Отвечает команда безопасности. Меняется только когда выходит CVE или новый минорный релиз.

**org-base** — первый слой корпоративного мира. Здесь ОС «отрывают от интернета»:
публичные репозитории заменяются на внутренние зеркала Artifactory.
Дальше `apt install` ходит только во внутреннюю сеть — никогда наружу.
Плюс DNS, NTP, CA, мониторинг. Провизионируется shell-скриптами — намеренно просто.

**org-golden** — образ для людей. Docker, kubectl, компиляторы, Python-окружения.
Здесь уже Ansible: роль `company.example` настраивает всё нужное под конкретную команду.

> *Три слоя — три вопроса: «Безопасно ли?», «Наше ли?», «Удобно ли?»*

## Как это работает: жизнь одного образа

### Шаг 1. vendor-base из ISO

Всё начинается с чистого ISO. Packer поднимает QEMU-машину, монтирует ISO,
передаёт `cloud-init` / `kickstart`-конфиг через встроенный HTTP-сервер и ждёт установки.
Потом диск упаковывается в `.box` через vagrant post-processor.

Поддерживаются два режима загрузки:

- **legacy BIOS** — для того, что ещё не выбросили;
- **UEFI** — для всего остального; требует `efivars.fd`.

Имя итогового файла несёт всю мета-информацию — без базы данных, без реестра:

```text
ubuntu-24.04-production-uefi-vendor-base-20260316.2212.box
^^^^^^^  ^^^^^  ^^^^^^^^^^  ^^^^  ^^^^^^^^^^^  ^^^^^^^^^^^^
  OS    version    env      boot    layer        timestamp
```

> *Имя файла — это документация, которую невозможно потерять отдельно от файла.*

### Шаг 2. org-base поверх vendor-base

Packer скачивает `vendor-base` из Artifactory, поднимает VM через Vagrant,
запускает shell-скрипты. Главный трюк: OS-специфичный скрипт удаляет все `.repo`/`sources.list`
и прописывает только внутренние зеркала. С этого момента VM «оторвана от интернета»
на уровне пакетного менеджера.

### Шаг 3. org-golden поверх org-base

Аналогично, но provisioner — **Ansible**. Packer сам открывает SSH к гостю
и гоняет playbook с хоста. Никакого `ansible-local`, никакого инвентаря вручную.

Можно иметь несколько вариантов golden-образа под разные команды (ML, backend, infra).

### Запуск: только make, никакого bash в руках

```bash
# Один слой
make build TYPE=vendor-base OS=ubuntu VERSION=24.04 ENV=staging
make build TYPE=org-base    OS=ubuntu VERSION=24.04 ENV=staging
make build TYPE=org-golden  OS=ubuntu VERSION=24.04 ENV=staging

# Вся цепочка (vendor-base → org-base → org-golden, legacy + UEFI)
make build-chain OS=ubuntu VERSION=24.04 ENV=staging

# RHEL, только production, только legacy
make build-chain OS=rhel VERSION=9.6 ENV=production UEFI=false
```

Chain-операции симметричны — если можно собрать, можно залить и удалить:

```bash
make build-chain   OS=ubuntu VERSION=24.04 ENV=staging
make upload-chain  OS=ubuntu VERSION=24.04 ENV=staging STORAGE=artifactory
make remove-chain  OS=ubuntu VERSION=24.04 ENV=staging
```

Повседневное:

```bash
make show-local-boxes    # что собрано локально
make show-remote-boxes   # что лежит в хранилище
make fetch   TYPE=org-golden  # забрать последний образ
make test    TYPE=org-golden  # smoke-тест
make ssh     TYPE=org-golden  # зайти внутрь box руками (для отладки и любопытства)
```

Три уровня очистки:

```bash
make clean-env     # убить stale libvirt-домены и vagrant-tmp
make clean-builds  # clean-env + удалить builds/ и logs/
make clean         # полная очистка включая .vagrant.d
```

## Среда сборки: матрёшка внутри матрёшки

Packer и Vagrant работают **внутри Docker-контейнера**, который монтирует
`/dev/kvm` и libvirt-сокет. То есть: Docker запускает контейнер, внутри которого
Packer поднимает KVM-виртуалку, внутри которой Vagrant тестирует образ.

> *Это не inception. Это DevOps.*

Зачем так:

1. **Воспроизводимость** — версии Packer/Vagrant/Ansible прибиты в `Dockerfile`.
   «А у меня другая версия» перестаёт быть аргументом.
2. **CI/CD** — GitHub Actions runner запускает тот же образ. Ноль расхождений.

```bash
docker run --privileged --device /dev/kvm \
  -v /var/lib/libvirt:/var/lib/libvirt:rw \
  -v /var/run/libvirt:/var/run/libvirt:rw \
  boxsmith-builder make build TYPE=vendor-base OS=ubuntu VERSION=24.04
```

**Место Ansible в этой схеме.** Ansible используется **только для `org-golden`**.
Packer поднимает гостевую VM через Vagrant, а затем запускает стандартный
`ansible` provisioner: Packer сам устанавливает SSH-соединение к гостю
и выполняет playbook с хоста — никакого `ansible-local`, никакого ручного inventory.

`org-base` провизионируется набором **shell-скриптов** (OS-специфичный скрипт + общие:
`sudoers.sh`, `fix_time.sh`, `sshd.sh`, `system.sh`, `setup.sh`).
Это намеренно: shell проще отлаживать на этом уровне, а Ansible подключается уже поверх
готовой корпоративной базы в `org-golden`.

## Smoke-тест: доверяй, но проверяй

Перед публикацией каждый `.box` проходит автотест:

1. Импортировать box в libvirt.
2. Поднять VM через `vagrant up`.
3. Зайти по SSH, прочитать `/etc/box-release`.
4. Проверить, что версия, слой и тайм-штамп совпадают.
5. Уничтожить VM.

Если тест упал — образ не публикуется. Автоматически. Без обсуждений.

> *QA — это не про то, чтобы найти баги. Это про то, чтобы их не выпустить.*

## Именование: имя файла — источник правды

```text
ubuntu-24.04-production-uefi-org-golden-20260316.2144.box
```

Python-парсер в `common.py` разбирает это в структуру:

```python
meta = parse_box_filename(box_path.name)
# → BoxMeta(os='ubuntu', version='24.04', env='production',
#           boot='uefi', box_type='org-golden', timestamp='20260316.2144')
```

Timestamp `YYYYMMDD.HHMM` — это и версия, и история, и причина для rollback.
Никакой базы данных. Никакого реестра. Просто `ls -ltr builds/`.

## Добавление и удаление ОС

«Нам нужен AlmaLinux 9.8» превращается из недельного эпика в:

```bash
# Добавить на основе rhel-9.6
make add-new-os OS=rhel VERSION=9.8 FROM=rhel-9.6

# Сначала посмотреть, что будет (DRY_RUN)
DRY_RUN=1 make add-new-os OS=rhel VERSION=9.8 FROM=rhel-9.6

# Убрать ОС из шаблонов
make remove-os OS=rhel VERSION=9.8
```

Под капотом: копируется директория шаблонов, заменяются версии в текстах,
генерируются json-файлы для org-base/org-golden, копируются Ansible vars.
Оба инструмента **идемпотентны** — не перезапишут файлы, изменённые вручную.

> *Хороший инструмент делает правильные вещи по умолчанию и не мешает делать нестандартные.*

## Хранение: где лежат коробки

Кузница кует — склад хранит. Поддерживаются два бэкенда:

```bash
# в Artifactory (внутри)
STORAGE=artifactory make upload TYPE=org-golden OS=ubuntu VERSION=24.04

# в S3 (если вы в облаке и не боитесь счётов)
STORAGE=s3 S3_BUCKET=my-boxes make upload TYPE=org-golden OS=ubuntu VERSION=24.04
```

Рядом с каждым `.box` создаётся `.meta.json` — маленький sidecar с SHA-256,
ссылкой на родительский образ (`based_on`) и тайм-штампом.
Это позволяет `fetch-box.py` найти нужный образ без мучительного `ls` по всему бакету вручную.

> *Склад без инвентаря — это не склад. Это куча.*

## CI/CD: два workflow и немного дисциплины

GitHub Actions, потому что бесплатно, достаточно функционально и иногда больно.

**`validate.yml`** — на каждый PR. Быстро, дёшево, без VM:

* `packer validate` для всех шаблонов;
* `ansible-lint` для playbooks;
* unit-тесты Python-скриптов.

Если что-то из этого падает — PR не мёрджится. Автоматически.
Никаких «ну там просто линтер ругался, это не важно».

**`build.yml`** — при мёрдже в `main` или ночью по расписанию:

* полная цепочка `build-chain` для всех поддерживаемых ОС;
* `upload-chain` в Artifactory;
* уведомление при ошибке.

> **CI/CD — это не про автоматизацию. Это про то, чтобы падал пайплайн, а не прод.**

## Что получилось

Честно:

* `make build` даёт **одинаковый результат** на любой машине с KVM — больше никакого «у меня работало»;
* новый разработчик поднимает полностью настроенную VM за `vagrant up` — **3–5 минут** вместо «я три часа ставил зависимости»;
* **rollback** — сменить тайм-штамп в имени box. Всё. Никакого Васи в Телеграме;
* добавить ОС — **одна команда** и PR с двумя файлами;
* менять org-base не ломает org-golden (если только вы специально не постараетесь).
